# Kubernetes это

Kubernetes - это оркестровщик, управляющий контейнерами с приложениями, в окружении для деплоинга.

*Основная цель - разделение разработки от монолитного к микросервисам, за которые отвечают различные контейнеры*.

### Актуальность темы

1. Тренд на микросервисы
2. Повышение использования контейнеров
3. Необходимость управления большим числом контейнеров

### Основные функции средств оркестрации

1. Высокая доступность, приложения всегда активны
2. Быстрое маштабирование
3. Возможность бекапа и восстановления контейнеров


# Архитектура

**Основной кластер состоит из:**

*Нода - ВМ или физическая машина*

1. Главная Нода
	- Рабочие ноды (Kubelet - kubernetes-процесс, позволяющий контейнерам обмениваться данными, выполнять задачи)

**В каждой ноде находятся контейнеры, задеплоенные туда**

Рабочая нода имеет параметр ON - говорящий о работе контейнеров

**Для главной ноды необходима активная работа K8s процессов**



## Архитектура главной ноды


**Главная нода состоит из**

1. API сервера - контейнер, где хранится окружение кластера K8 (UI, API, CLI) `api`
2. Controller manager - контейнер, хранит таймлайн происходящего в кластере `c-m` (Логирование)
3. Scheduler - контейнер-планировщик, определяющий, в какой ноде запустится очередной контейнер `sched`
4. etcd - Резервное хранилище Kubernetes `etcd`
## Архитектура рабочей ноды

**Рабочая нода состоит из:**

1. Docker контейнера
2. kubelet процесса

**Также создается виртуальная сеть - единая машина для работы всего кластера**


# Компоненты kubernetes

### 1. Pod

**Pod** - *абстракция, которая находится над контейнером. Pod - наименьшая единица развертывания в kubernetes.*

Зачастую pod имеет один контейнер под собой.
Каждый pod имеет свой публичный IP-адрес
Pod - эфемерный компонент, который удаляется, если контейнер завершает работу.

### 2. Service

**Service** - *статический IP-адрес, который может быть применен к каждому pod, а также служит для взаимодействия между подами*

Время работы Serivce и pod не связаны.

Для того, чтобы приложение было доступно в браузере,(внешний сервис) необходимо обратиться по сокету ноды.

### 3. Ingress

**Ingress** - *объект API, который управляет входным трафиком вне кластера к services ноды.*

Ingress маршрутизирует внешний https трафик к нужным сервисам

### 4. ConfigMap

**ConfigMap** - *внешний конфиг приложения, где хранятся например URL БД*

!! ConfigMap - только для публичной информации, так как ConfigMap не является защищенным.

### 5. Secret

**Secret** - *объект кластера, для безопасного хранения конфиденциальных данных.*

!! *Kubernetes secrets по умолчанию хранятся в etcd сервера в незашифрованном виде*

Данные из secret можно использовать в контейнере ноды.

### 6. Volume

**Volume** - *объект kubernetes, который связан с внешними дисками на самой машине, где развернуто приложение, хранилища volume находятся вне K8 кластера*

Volume служит для хранения БД и для их бекапа.

Volume не предусматривает автоматического сохранения данных на своих хранилищах.

### 7. Deployment

**Deployment** - *высокоуровневый объект, описывающий желаемое поведение группы подов и управляющий их жизненным циклом.*


### 8. STS

**STS (StatefulSet)** - *объект kubernetes, который связывает БД из нод с контейнерами приложений с основным хранилищем*



# Правила безопасного деплоинга приложений

1. Каждый контейнер с приложением должен храниться на нескольких нодах __с одинаковым IP адресом__, чтобы в случае падения приложения в 1 ноде, пользователь имел доступ к приложению, которое развернуто на 2 ноде.
2. Для каждого приложения должен существовать балансировщик трафика, котоый распределяет трафик по нескольким нодам.
3. Для каждого приложения должен быть определен блюпринт. (Определяет количество подов приложения)
4. БД, расположенные с приложениями в 1 ноде должны иметь доступ к одному и тому-же хранилищу, чтобы всегда иметь актуальные данные без падения приложения при отключении БД приложения


Альтернативы



# 1. Nomad

**Nomad** — один из самых близких концептуальных конкурентов Kubernetes. Это scheduler и orchestrator для контейнеров, бинарников и batch-задач, который может работать в cloud, on-prem и гибридных окружениях.
# 2. Docker Swarm

**Docker Swarm mode** встроен в Docker Engine и позволяет управлять кластером Docker-нод как swarm-кластером. Docker прямо описывает Swarm mode как механизм для управления кластером Docker daemons и отмечает, что текущие версии Docker включают Swarm mode; при этом старый Docker Classic Swarm больше активно не развивается.

# 3. Amazon ECS + AWS Fargate


**Amazon ECS** — fully managed container orchestration service от AWS для деплоя, управления и масштабирования контейнеризированных приложений. Он глубоко интегрирован с AWS-экосистемой.  
**AWS Fargate** позволяет запускать контейнеры через ECS без управления серверами или EC2-кластерами
